## 二、摘要

随着智能家居设备数量的持续增长，自动化规则配置已成为提升家庭场景智能化水平的重要方式。然而，现有智能家居自动化规则通常依赖 YAML 配置、图形化编排界面或厂商专有应用，存在学习成本高、表达不直观、跨平台迁移困难等问题。相比之下，中文自然语言描述更符合普通用户的使用习惯，但将中文规则直接映射为可执行配置时，往往会出现结构不稳定、字段缺失、设备实体识别错误以及规则逻辑冲突等问题，导致生成结果难以直接落地执行。针对上述问题，本文设计并实现了一种面向中文描述的智能家居自动化规则生成系统，提出了“中文规则描述—中间 DSL—规则校验与自动修复—YAML 转换—解释与可视化”的处理链路，以提升规则生成的结构化程度、合法性与可执行性。

在方法设计上，本文首先构建了面向智能家居自动化场景的第一版领域专用语言（Domain Specific Language, DSL），该 DSL 包含规则名称、触发器、条件、动作和执行模式等核心字段，支持时间触发、状态变化触发和事件触发三类触发方式，以及状态条件、时间范围条件和星期条件等常见约束形式。在此基础上，系统通过解析模块将中文规则描述转换为 DSL 草案，并结合设备知识库对规则中的实体、服务与逻辑关系进行校验；针对常见的确定性错误，如设备实体拼写错误、动作冲突、时间与星期格式错误、模式字段缺失等，进一步引入自动修复机制，提高规则的工程可用性。随后，系统将合法 DSL 转换为 Home Assistant 风格 YAML，并生成中文解释文本与 Mermaid 结构图，用于增强规则的可理解性与可维护性。

在实验部分，本文构建了包含 5000 条设备实体的智能家居设备知识库、5000 条中文规则测试样本以及 1800 条 gold DSL 标注数据，用于评估系统的工程稳定性与结构化预测质量。实验结果表明，相较于直接从中文生成 YAML 的基线方案，引入 DSL 中间层后，规则完整率由 88.20% 提升至 100.00%，校验通过率由 70.92% 提升至 100.00%，端到端可执行率由 70.92% 提升至 100.00%，说明中间 DSL 在提升规则结构完整性与执行稳定性方面具有显著效果。在规则自校验与自动修复实验中，系统的规则通过率由 30.06% 提升至 99.17%，自动修复增益达到 69.10%，表明“校验 + 局部修复”的组合能够显著增强规则生成系统的鲁棒性。在结构化预测质量评测中，系统在 1800 条 gold DSL 数据上的字段级准确率达到 0.5591，Trigger Type F1 达到 0.9569，Entity Selection Micro-F1 达到 0.8060，说明系统在规则触发类型识别方面较为稳定，但在大规模设备知识库上的实体 grounding 仍存在进一步优化空间。

综合来看，本文提出的基于中间 DSL 的中文智能家居自动化规则生成方案，能够有效缓解中文规则直接生成可执行配置时的结构不稳定与可执行性不足问题，在工程稳定性、自动修复能力和结构化表达方面具有较好的实验效果。本文工作可为后续中文智能家居规则生成系统的进一步研究与落地应用提供可复用的系统原型与实验基础。

**关键词：** 智能家居；自动化规则生成；领域专用语言；规则校验；自动修复；结构化表示

## 第1章 绪论

### 1.1 研究背景

本节主要说明智能家居设备和自动化规则在家庭场景中的应用背景。可以从智能家居设备数量持续增长、家庭自动化需求不断增强切入，指出当前规则配置方式仍主要依赖 YAML 配置文件、平台图形化界面或厂商封闭应用，这些方式普遍存在门槛较高、学习成本较大、表达能力有限以及跨平台复用困难等问题。随后引出中文自然语言规则输入的现实价值，即其更符合普通用户表达习惯，更容易被接受，但直接从中文映射为可执行规则又面临结构不稳定和语义缺失等困难。

### 1.2 研究问题

本节聚焦于两个核心问题。第一，如何将中文规则描述稳定地转化为结构化且可执行的智能家居自动化规则。第二，如何在规则生成后通过校验与自动修复机制提升规则的合法性、可执行性和整体稳健性。这里要强调，本文关注的不仅是“生成出来”，而是“生成后能否稳定落地执行”。

### 1.3 研究思路

本节介绍论文的总体思路。首先引入中间 DSL 作为中文规则与 YAML 配置之间的语义承接层，用于约束规则结构并支持后续处理；其次通过设备知识库、规则校验与自动修复机制增强工程可用性；最后通过中文解释与结构化可视化提升规则的可理解性。可以在本节给出简洁的问题求解流程图。

### 1.4 研究内容与创新点

本节建议从以下几个方面组织：

1. 设计适用于中文智能家居规则表达的第一版 DSL。
2. 构建从中文输入到 DSL、从 DSL 到 YAML 的完整规则处理流程。
3. 实现基于设备知识库的规则校验与自动修复机制。
4. 设计“直接 YAML 生成”与“DSL 中间层方案”的对比实验。
5. 引入解释与结构化可视化模块，提高规则可理解性。

创新点可以归纳为三个方面：

1. 提出中文智能家居规则生成中的中间 DSL 表示方案。
2. 将规则校验与自动修复纳入完整链路，增强最终可执行率。
3. 在工程实验与结构化指标两个层面验证系统有效性。

## 第5章 规则生成与规则处理方法

### 5.1 中文规则到 DSL 草案生成

本节介绍 parser 的基本流程：

- 使用提示词模板描述 DSL schema
- 注入设备实体知识库
- 调用可替换 llm_client 接口生成 DSL JSON 草案
- 进行 JSON 提取和解析失败重试

同时说明 mock_llm_client 在当前本地实验中的作用，即用于在无真实 API 环境下模拟系统行为。

### 5.2 规则校验机制

本节系统介绍 validator 的校验项：

- schema 合法性
- entity 是否存在
- service 是否被该 entity 支持
- actions 是否为空
- 同一设备是否存在明显冲突动作
- time / weekday / time_range 等字段是否合法

这一节要突出“规则生成”与“规则可执行”之间并不等价，必须依赖规则校验机制完成约束过滤。

### 5.3 自动修复机制

本节介绍 repairer 的设计与修复范围：

- mode 缺失时默认补全为 `single`
- entity 拼写错误时使用 RapidFuzz 进行近似匹配修复
- 时间与 weekday 字段格式纠正
- 冲突动作局部删除或调整
- service 前缀与实体类型不一致时给出建议或自动修复

本节需要强调：自动修复只处理“可确定性错误”，而不处理高歧义语义问题，以避免系统胡乱修改规则。

### 5.4 YAML 转换方法

本节介绍从 RuleDSL 到 Home Assistant 风格 YAML 的映射方式，包括 trigger、condition 和 action 的对应关系。重点说明 DSL 将前端中文语义与后端 YAML 配置解耦。

### 5.5 中文解释与结构可视化

本节介绍 explainer 与 visualizer 的作用。解释模块基于模板生成自然语言说明，可视化模块输出文本树与 Mermaid 图，帮助用户和开发者理解规则结构。该部分可以作为系统易用性与可维护性的辅助能力。

- 完成了中文智能家居自动化规则生成系统原型设计与实现
- 提出了中间 DSL 表示方法
- 构建了规则校验与自动修复机制
- 实现了 YAML 转换、中文解释与可视化输出
- 通过实验验证了方案有效性

### 8.2 主要结论

可归纳为三点：

1. DSL 中间层能够显著提升规则完整性、合法性与可执行性。
2. 规则校验与自动修复机制能够显著提升最终规则通过率。
3. 当前系统在结构化规则生成方面已具备一定能力，但实体 grounding 仍需进一步优化。

### 8.3 存在不足

建议从以下方面总结不足：

- gold 标注集仍非完全人工审校
- 当前解析侧仍使用 mock_llm_client，不代表真实大模型水平
- 长尾设备类型样本仍然偏少
- 尚未与真实 Home Assistant 平台进行实际联调

### 8.4 后续展望

可从四个方向展开：

1. 构建人工审校 gold 数据集，提高机器学习指标可信度
2. 接入真实 LLM，提高中文语义解析质量
3. 引入实体检索与排序指标，如 `Hit@K`、`MRR`
4. 与真实智能家居平台联调，验证系统在真实场景中的可部署性

1. 正文术语要统一使用“中间 DSL”“规则校验”“自动修复”“结构化指标”等表述。
